## WCAG 2.2 AA — accessibility requirements (NON-NEGOTIABLE)

This product serves vulnerable users — renters with disabilities, English as a second language, low literacy, screen reader users, keyboard-only users. Accessibility is not a polish layer. It's a requirement. Every item below must be verified before declaring Round 7 complete.

### Perceivable

- **Colour contrast minimums:**
  - Normal text (under 18.66px regular or 24px bold): minimum **4.5:1** ratio
  - Large text: minimum **3:1**
  - UI component boundaries (button borders, input borders): minimum **3:1**
  - Verify ink `#171717` on bone `#FBF8F1` (passes — ~16:1)
  - Verify ink-muted `#6B6B6B` on bone `#FBF8F1` (passes — ~5.7:1)
  - Verify sage `#5A8B7A` eyebrows on bone — verify it passes 4.5:1 minimum, if not, darken to `#3D5F50`
  - Verify pale-blue `#B8C8D9` on ink-blue `#1E3A5F` for dark cards
  - Verify clay/wine severity badges have sufficient contrast against their backgrounds

- **Colour is never the only indicator:** Severity badges have BOTH a colour AND a text label ("HIGH" / "MEDIUM" / "LOW"). Filter pill active state has both colour change AND a visual marker. Sage dot pulse has an accompanying aria-live region announcing state changes if applicable.

- **Alt text:** Every meaningful image has descriptive alt text. Decorative-only SVGs use `aria-hidden="true"` or empty alt `alt=""`. The hero illustration has alt like `"Document with magnifying glass — Check the Lease analyses your lease."`

- **Text resizing:** Page works at 200% browser zoom without horizontal scroll or content cut-off.

### Operable

- **Keyboard navigation:** Every interactive element reachable via Tab in logical order matching visual order. Test the full flow without a mouse: nav → hero CTAs → state pills → feature cards → upload → results.

- **Focus states:** Every focusable element has a visible focus ring. Use a 2px solid ink-blue outline with 2px offset. NOT the default browser blue. Apply globally:
```css
*:focus-visible {
  outline: 2px solid #1E3A5F;
  outline-offset: 2px;
  border-radius: 4px;
}
```

- **Skip-to-main-content link:** First focusable element on every page. Visually hidden until focused, then appears top-left.
```tsx
<a href="#main" className="sr-only focus:not-sr-only focus:fixed focus:top-2 focus:left-2 focus:bg-ink-blue focus:text-bone focus:px-4 focus:py-2 focus:rounded">
  Skip to main content
</a>
```

- **Touch targets:** Minimum 44×44px for ALL interactive elements (buttons, links, pills, checkboxes, radio cards). Pad smaller visual elements with invisible touch padding if needed.

- **No keyboard traps:** Modal dialogs, expanded accordions, dropdown menus must be escapable via Esc key.

- **Time limits:** No auto-redirects without user control. Loading state has no timeout.

### Understandable

- **`lang="en"` on `<html>`** declared.

- **Form labels:** Every input has a visible label using proper `<label htmlFor="...">`. Placeholder text is NEVER the only label.

- **Error messages:** Specific, constructive, near the field. Not just red borders — actual text explaining what's wrong and how to fix it.

- **Reading level:** All copy at 6th-grade level minimum. Test with hemingwayapp.com or similar. Replace legal jargon with plain English everywhere except direct statute citations.

- **Predictable navigation:** Same nav, same footer, same brand presentation on every page.

### Robust

- **Semantic HTML:** Use `<nav>`, `<main>`, `<section>`, `<article>`, `<footer>`, `<button>` (not `<div onClick>`), `<a>` for links (not `<button>` for navigation).

- **ARIA live regions** for dynamic content:
  - Loading state announces to screen readers: `aria-live="polite"` on the loading card, with the rotating message updating
  - Analysis complete announces: `aria-live="polite"` on the results region
  - Error states announce: `aria-live="assertive"` for failures

- **Heading hierarchy:** One `<h1>` per page (the H1 in hero). All other headings follow logical nesting (H2 for sections, H3 for cards, H4 for sub-elements).

- **Icon-only buttons** have `aria-label`:
```tsx
<button aria-label="Remove file"><X /></button>
<button aria-label="Dismiss banner"><X /></button>
```

- **Buttons vs links:** Use `<button>` for actions, `<a href>` for navigation. Never confuse them.

### Tools to verify

After building, run these checks (Agent should attempt these via the Replit shell):

1. **Lighthouse Accessibility audit** — must score 95+. Run in dev tools or via `npx @lhci/cli@0.13.x autorun`.
2. **axe DevTools** — paste a small script that uses `@axe-core/react` to log issues, OR install the browser extension and run on the deployed page.

4. **Manual screen reader test:** Optional but ideal — use VoiceOver (Cmd+F5 on Mac) or NVDA on the landing page. Hero, CTA, sections, feature cards should all announce sensibly.

### Specific Check the Lease accessibility additions

- The `<svg>` magnifying-glass animation in the hero illustration must NOT auto-play if `prefers-reduced-motion: reduce`. Wrap the animation in a media query.
- The marquee curving italic divider must NOT scroll if `prefers-reduced-motion: reduce` — render static.
- The sticky scroll-progress bar must NOT animate position changes — purely declarative width based on scroll.
- The interactive state preview pills must be `<button>` elements with `aria-pressed` reflecting active state.
- The Without/With comparison cards must NOT rely on colour alone — sage vs wine — to convey meaning. Each has a clear text label ("WITHOUT US" / "WITH US") and ✓ / · symbols on the bullet items.
- Loading state's rotating messages should announce via `aria-live="polite"` so blind users know analysis is in progress.

### 1. Sticky scroll-progress bar

Thin (2-3px) bar at the top of the viewport showing scroll progress through the page. Subtle ink-blue fill (`#1E3A5F`). On the landing page only. Below the nav, above content.

```tsx
// In your main layout, after the nav:
const [scrollProgress, setScrollProgress] = useState(0);
useEffect(() => {
  const onScroll = () => {
    const docHeight = document.documentElement.scrollHeight - window.innerHeight;
    const progress = (window.scrollY / docHeight) * 100;
    setScrollProgress(progress);
  };
  window.addEventListener('scroll', onScroll, { passive: true });
  return () => window.removeEventListener('scroll', onScroll);
}, []);
```

## Animation & motion language

Every element gets motion intentionally. Use this as the spec:

| Element | Motion |
|---|---|
| Hero text | Fade-up 200ms ease-out, staggered (badge → H1 → sub → CTAs → stats) |
| Section reveals on scroll | Fade-up 400ms ease-out, triggered when 20% of section enters viewport (use IntersectionObserver) |
| Buttons (primary + ghost) | Hover: scale(0.97) + colour wash, 150ms ease |
| Cards | Hover: scale(0.99) + border tightens to medium opacity, 200ms ease |
| Links | Hover: text-underline-offset 3px → 4px, 150ms ease |
| Marquee curving text | Slow horizontal scroll via animated startOffset, 30s linear infinite |
| Sage pulse dot in nav badge | Existing scale animation, keep |
| Document magnify (hero illustration) | Existing pulse, keep |
| Page transitions | Fade in main on route change, 200ms ease-in |

CRITICAL: respect `prefers-reduced-motion`. Wrap all animations in `@media (prefers-reduced-motion: no-preference)` or disable them entirely if the user has reduced motion set.
